# AGENTS.md — Upwork Freelance Automation Suite

> This file is written for AI coding agents. It assumes you know nothing about the project and need to work on it safely. Read this before editing, building, or running anything.

---

## 1. Project overview

The **Upwork Freelance Automation Suite** is a personal productivity system that helps a single freelancer (Yogesh Mistry) discover relevant Upwork gigs and draft tailored proposals. **It never submits proposals automatically** — the user copies generated proposal text and pastes it into Upwork manually.

The system is currently in **Sprint 0/1**. The design, schema, agent prompts, session helpers, and scoring logic are written, but the core scraper and MCP server tools are stubs or missing. The repository is intentionally generic: it contains no hardcoded personal data, credentials, or Upwork session state.

There are three user-facing agents:

| Command | Source file | Purpose |
|---------|-------------|---------|
| `/setup` | `agents/upwork-setup.md` | Interview-style onboarding that produces the user's knowledge base and config files |
| `/find-gigs` | `agents/upwork-gig-finder.md` | Runs Playwright searches, scores gigs, inserts new ones to Supabase, shows a digest |
| `/write-proposal` | `agents/upwork-proposal-writer.md` | Generates a ready-to-paste proposal for a given gig URL or gig ID |

The canonical product spec is `PRD.md`. Implementation details and architecture notes live in `BUILD_NOTES.md`. `CLAUDE.md` contains the original Claude Code project configuration.

## 2. Technology stack

| Layer | Technology | Notes |
|-------|------------|-------|
| Runtime | Node.js 20+ | Project uses CommonJS (`"type": "commonjs"`) |
| Browser automation | Playwright 1.44+ | Headed browsing only; read-only scraping, no form submission |
| Database | Supabase (PostgreSQL) | `mcp-server/schema.sql` defines three tables: `upwork_gigs`, `upwork_proposals`, `upwork_contracts` |
| Database client | `@supabase/supabase-js` | Called from Node scripts and planned for MCP tools |
| MCP server | Supabase Edge Functions (Deno/TypeScript) | `mcp-server/index.ts` and `mcp-server/tools/*.ts`; currently stubs |
| Agent orchestration | Claude Code / Cowork | Agent prompts are plain Markdown files invoked via `claude agents/<file>.md` |
| Config / knowledge base | Markdown + JS mirror | `USER_CONFIG.md` and `playwright/user-config.js` |
| OS notifications | macOS `osascript` | Used by `playwright/check-session.js` when the session expires |

Only two runtime dependencies are declared in `package.json`: Playwright and Supabase. TypeScript is listed as a dev dependency for the MCP tooling.

## 4. Build and runtime commands

All commands run from the project root.

### Install dependencies

```bash
npm install
```

Requires Node.js 20 or higher. There is no `package-lock.json` in the repo yet.

### Run the setup interview

```bash
claude agents/upwork-setup.md
```

This must be run before any other agent. It creates `USER_CONFIG.md`, `playwright/user-config.js`, `MASTER_PROFILE.md`, `MASTER_PROPOSALS.md`, and `MASTER_SKILLS.md`.

### Save the Upwork session

```bash
node playwright/save-session.js
```

Opens a headed browser. Log in manually (including 2FA). The session is saved to `playwright/session.json`.

### Check session health

```bash
node playwright/check-session.js
```

Exits `0` if the session is valid, `1` and sends a macOS notification if re-authentication is needed.

### Run the daily gig search

```bash
claude agents/upwork-gig-finder.md
```

Also aliased in `package.json` as `npm run find-gigs`.

> **Current state:** This command depends on `playwright/scraper.js`, which does not exist yet. The scoring logic in `playwright/scorer.js` is implemented and can be unit-tested independently.

### Generate a proposal

```bash
claude agents/upwork-proposal-writer.md
```

Or via the package script: `npm run write-proposal`.

### Direct Node scripts

| Script | Command | Status |
|--------|---------|--------|
| Save session | `npm run save-session` / `node playwright/save-session.js` | Implemented |
| Check session | `npm run check-session` / `node playwright/check-session.js` | Implemented |
| Scrape gigs | `npm run scrape` / `node playwright/scraper.js` | **Not implemented** |

## 5. Code style guidelines

### JavaScript (Playwright helpers)

- Use **CommonJS** (`require`/`module.exports`) because `package.json` sets `"type": "commonjs"`.
- Use `async/await` for asynchronous Playwright code.
- Add a short JSDoc-style header comment explaining each module's purpose.
- Export functions via `module.exports = { ... }`.
- Prefer semantic, role-based selectors for structural checks (see `canary.js`) and `data-test` selectors for data extraction (centralised in `selectors.js`).
- Parse DOM strings defensively; assume values may be `undefined` or oddly formatted.

### TypeScript (MCP server)

- Each tool lives in its own file under `mcp-server/tools/`.
- Every tool stub currently exports one async function that takes `payload: unknown`.
- Follow the payload signatures in `BUILD_NOTES.md` section "MCP Tool Signatures" exactly.

### Agent prompt files (`agents/*.md`)

- Write in second-person instructions directed at the agent.
- Keep rules explicit and numbered.
- Do not hardcode user-specific values (queries, keywords, rates) inside agent files; reference `USER_CONFIG.md` or `playwright/user-config.js` instead.

### Logging

- Accept an optional `logger` object (`{ info, warn, error }`) in helper functions so the caller can inject structured logging.
- Fall back to `console.*` when no logger is provided.
- Include ISO timestamps in standalone script output when practical.

## 6. Testing instructions

There are no automated tests in the repo yet. Use the manual test plan from `BUILD_NOTES.md`:

1. **Setup agent test:** Run `/setup`, verify all five generated files are created and populated.
2. **Scorer unit test:** Import `playwright/scorer.js` and call `scoreGig(...)` with sample gig objects. Verify score ranges 0-100 and priority maps correctly.
3. **Session test:** Run `node playwright/save-session.js`, then `node playwright/check-session.js` — expect exit code 0.
4. **Scraper test (once implemented):** Run a single search query, verify at least 5 results are returned and validation logic skips malformed cards.
5. **Proposal generation test:** Generate proposals for 3 real gigs and review quality, word count, and honesty.
6. **End-to-end test:** Run the full pipeline on 1 real gig before enabling scheduled runs.

### Quick scorer sanity check

You can run a one-off Node check without a browser:

```javascript
const { scoreGig } = require('./playwright/scorer');
const sample = {
  title: 'Microsoft 365 Copilot training for customer support team',
  description: 'Need an AI adoption consultant to run workshops...',
  budget: '$60.00/hr',
  paymentVerified: true,
  clientRating: '4.93 of 5',
  postedAt: '2 hours ago',
  proposalCount: '12'
};
console.log(scoreGig(sample));
```

> Note: `user-config.js` currently contains placeholder arrays, so keyword scoring will not fire until `/setup` is run or the arrays are populated manually.

## 7. Security and compliance considerations

### Hard constraints

- **Read-only Playwright.** Never fill a form, click a submit button, or perform any action that changes state on Upwork.
- **Manual submission only.** Generated proposals are plain text for the user to paste manually.
- **No fabricated credentials.** Agents must only claim skills and outcomes present in `MASTER_SKILLS.md` and `MASTER_PROPOSALS.md`.
- **No fake reviews, no account sharing, no automated Upwork actions.**

### Secrets

- `UPWORK_SUPABASE_URL` and `UPWORK_SUPABASE_KEY` must be set in `.env`.
- `.env` is gitignored. Never commit it.
- `playwright/session.json` contains Upwork session cookies and is gitignored.
- `playwright/state.json` is gitignored.
- No Upwork password or credentials are stored in code.

### Bot-detection mitigations

The Playwright layer intentionally uses human-like behaviour:

- Headed browser (`headless: false`) for scraping.
- Realistic viewport (`1440x900`) and Chrome user agent.
- `waitUntil: 'domcontentloaded'` instead of `networkidle`.
- Simulated scroll and randomised delays before extracting data.
- Max 3 search queries per session, 25–40 minute gaps between sessions, max 10 queries per calendar day.
- Session cookies are persisted back to `session.json` after each run.

## 8. Database schema overview

Run `mcp-server/schema.sql` in the Supabase SQL editor before using the system.

Three tables:

- `upwork_gigs` — discovered gigs with scoring, priority, and status.
- `upwork_proposals` — generated proposals, bid info, connects spent, screening answers.
- `upwork_contracts` — active/completed contracts, earnings, review status.

Key columns:

- `upwork_gigs.url` has a `UNIQUE` constraint for deduplication.
- `upwork_gigs.status` and `upwork_proposals.outcome` are PostgreSQL enums.
- `created_at` and `updated_at` columns auto-update via triggers.
- Row Level Security is enabled; service role policies allow full access for the Edge Function.

## 10. Development workflow

1. Read `PRD.md` for the requirement being implemented.
2. Read `BUILD_NOTES.md` for architecture, tool signatures, and anti-bot details.
3. If changing Playwright selectors, edit `playwright/selectors.js` only — do not scatter selectors across files.
4. If changing scoring weights or thresholds, keep keyword lists in `playwright/user-config.js` and the scoring math in `playwright/scorer.js`.
5. If implementing MCP tools, follow the exact signatures in `BUILD_NOTES.md` section "MCP Tool Signatures".
6. Keep the read-only, manual-submission, no-fabrication constraints in mind for every change.
7. Update this `AGENTS.md` if you change conventions, commands, or architecture.
